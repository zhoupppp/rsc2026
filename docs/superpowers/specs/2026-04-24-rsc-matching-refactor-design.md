# RSC 数据导入与多维度交叉比对重构设计

## 1. 背景与目标
为了支持后续“金融人物库网站”的扩展需求，并解决现有比对脚本仅支持旧版简化 Excel 的问题，本次重构旨在：
1. **支持复杂多表 Excel**：读取包含多个工作表、结构更复杂的最新 RSC 用户及机构数据。
2. **数据入库解耦**：将全量用户与机构的“扩展数据”持久化存储到 SQLite 数据库中，以供将来网站全局使用。
3. **提升匹配准确率**：利用新数据中机构的【全称、简称、别名/曾用名】等多维度字段，配合层级匹配算法，大幅提高自动匹配成功率，极大降低人工核对（Needs Review）的数据量。
4. **兼容现有系统**：保持原有 `rsc_user_mapping` 的表结构不变，确保现有的前后端逻辑及大屏监控不发?. **兼容现有系统**：保持原有 `rsc_user_mapping` 的表结构??脚本 (`scripts/import_rsc_data.py`)
该脚本专门负责读取新版 RSC Excel 文件，并将各工作表数据解析写入 `financial_data.db` 的新表中。
- **数据源**：
  - `RSC用户：完整信息...xlsx` (核心主键: `uid`)
  - `RSC机构库：完整信息...xlsx` (核心主键: `机构OID`)
- **新增数据表 (SQLite)**：
  - `rsc_users` (核心用户表：uid, 姓名, 认证类型, 机构OID, 职位...)
  - `rsc_user_profiles`, `rsc_user_behaviors`, `rsc_user_industry` (用户扩展信息表)
  - `rsc_orgs` (核心机构表：oid, 全称, 简称, 别名/曾用名...)
  - `rsc_org_profiles` 等 (机构扩展信息表)
- **处理策略**：采用全量 UPSERT 模式，即主键存在则更新，不存在则插入。

### 2.2 升级后的核心比对脚本 (`scripts/rsc_cross_check.py`)
该脚本不再读取 Excel，而是直接从上述构建好的 `rsc_users` 和 `rsc_orgs` 表中提取待比对人员及其多维度的机构名称集合，然后与 SAC/AMAC 的爬虫数据进行比对。
- **多维度层级匹配算法**：
  1. **数据清洗**：剥离 SAC/AMAC 机构名称以及所有 RSC 机构名称（全称、简称、曾用名）的常见后缀（如“股份有限公司”、“证券营业部”等）。
  2. **精确匹配 (Level 1)**：清洗后的爬虫机构名与任一 RSC 机构名完全一致，判定为**高置信度匹配**。
  3. **包含匹配 (Level 2)**：清洗后的爬虫机构名被包含于 RSC 机构名中，或反之（且长度差合理），判定为**高置信度匹配**。
  4. **高相似度模糊匹配 (Level 3)**：利用 `SequenceMatcher` 计算相似度，若 `> 85%`，判定为**高置信度匹配**。
  5. **临界需确认匹配 (Level 4)**：相似度在 `60% ~ 85%` 之间，判定为**需人工确认**，不自动入库。
- **“待更新” (Outdated) 逻辑**：与原有逻辑一致，若匹配成功但爬虫数据中的最新任职机构与 RSC 机构不符或已离职，标记为 `is_outdated = True`。

### 2.3 输出产物
1. **数据库更新**：仅将“高置信度”匹配成功的记录 UPSERT 到原有的 `rsc_user_mapping` 表。
2. **报表生成**（供运营核对）：
   - `outdated_rsc_users.xlsx`：高置信度匹配成功，但已跳槽/离职的待更新用户。
   - `needs_review_rsc_users.xlsx`：临界值（60%-85%）用户，需人工手动判断。
   - `unmatched_rsc_users.xlsx`：查无此人或机构完全不匹配的用户。

## 3. 风险控制与兼容性
- 核心映射表 `rsc_user_mapping` 结构完全不动，现有的 `backend/main.py` 左连接查询和前端展示均能无缝衔接。
- 后续的“金融人物库”只需基于 `uid` 直接查询 `rsc_users` 及其关联扩展表即可获得全量画像，不会与 SAC/AMAC 爬虫核心数据产生耦合冲突。
